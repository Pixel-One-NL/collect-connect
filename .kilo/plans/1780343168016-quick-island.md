# Fix LDraw ZIP Download Timeout + Stream-Attach (bound memory end to end)

## Problem

Running `php artisan part:import-color-images --full` fails with:

```
Illuminate\Http\Client\ConnectionException
cURL error 28: Operation timed out after 30001 milliseconds with 419876020
out of 708825624 bytes received ... for https://cdn.rebrickable.com/media/dow...
```

Some per-color LDraw archives are very large (~700 MB). The current downloader
in `RebrickableDownloadService::retrieveZipEntriesFromUrl()` uses:

```php
$response = Http::get($url);          // buffers the ENTIRE body in memory
...
fwrite($tempZip, $response->body());  // whole body again as a PHP string
```

Two defects:

1. **Default 30s timeout** — Laravel's HTTP client defaults to a 30-second
   overall timeout. A 700 MB transfer cannot complete in 30s, so cURL aborts
   with error 28.
2. **Full in-memory buffering** — `Http::get()` + `$response->body()` loads the
   entire archive into PHP memory before it is written to a temp file. Even if
   the timeout were raised, a ~700 MB string risks exhausting `memory_limit`.

## Goal

Download large LDraw ZIP archives reliably by **streaming the response body
directly to a temp file on disk** (no full in-memory buffer) with a **generous
overall timeout** and a **short connect timeout**, both configurable.

## Decisions (confirmed with user)

- **Timeout strategy:** generous overall timeout (default 600s) + short
  `connectTimeout` (default 15s), so a dead/unreachable host fails fast while a
  slow large transfer is allowed to finish. Both configurable via
  `config/rebrickable.php` (env-overridable).
- **Job retry/resilience:** NO changes. Do not add `$tries`/`backoff()`/job
  `$timeout`, and do not change when `ldraw_images_imported_at` is stamped. Keep
  current behavior (job fails + is logged on unrecoverable error; 404 still
  stamps the timestamp via the empty-generator path).
- **Scope:** fix `retrieveZipEntriesFromUrl()` (download streaming) AND refactor
  `ImportColorLdrawImagesJob` to attach images in bounded batches as it streams
  (so the whole color's PNGs are never held in memory at once). Leave the CSV
  method (`retrieveRebrickableDataFromUrl()`) untouched in this change.
- **Job memory:** confirmed to be folded into this plan — replace the
  "buffer-all-then-attach" approach with chunked stream-attach.

### 2. Stream the download to disk in `retrieveZipEntriesFromUrl()`

File: `app/Domain/Rebrickable/Services/RebrickableDownloadService.php`

Replace the in-memory `Http::get($url)` + `fwrite($response->body())` approach
with a streamed download using the HTTP client's `sink()` (writes the response
body straight to a file path/stream as it arrives) plus explicit timeouts.

Key points for the rewrite of `retrieveZipEntriesFromUrl()`:

- Create the temp file first and resolve its path (keep the `tmpfile()` handle
  in scope for the whole generator so the OS does not delete it early):

  ```php
  $tempZip = tmpfile();
  $zipPath = data_get(stream_get_meta_data($tempZip), 'uri');

  if (! $zipPath) {
      return;
  }
  ```

- Stream the body to that path with timeouts:

  ```php
  $response = Http::connectTimeout((int) config('rebrickable.ldraw_connect_timeout'))
      ->timeout((int) config('rebrickable.ldraw_download_timeout'))
      ->sink($zipPath)
      ->get($url);

  if ($response->failed()) {
      return; // A missing archive (e.g. 404) is expected for some colors.
  }
  ```

  Using `->sink($zipPath)` streams bytes to disk instead of holding them in
  `$response->body()`, eliminating the large-string memory spike. The longer
  `timeout()` allows the full transfer; the short `connectTimeout()` keeps a
  dead host from hanging for the full window.

- Open and iterate the archive exactly as today (this part is already correct):

  ```php
  $zip = new ZipArchive;

  if ($zip->open($zipPath) !== true) {
      return;
  }

  for ($i = 0; $i < $zip->numFiles; $i++) {
      $name = $zip->getNameIndex($i);

      if (! $name || str_ends_with($name, '/')) {
          continue;
      }

      $stream = $zip->getStream($name);

      if (! $stream) {
          continue;
      }

      $contents = stream_get_contents($stream);
      fclose($stream);

      if ($contents === false) {
          continue;
      }

      yield $name => $contents;
  }

  $zip->close();
  ```

- Keep `$tempZip` referenced until after `$zip->close()` so the temp file is not
  garbage-collected/removed mid-iteration. (It is auto-removed when the handle
  goes out of scope at generator completion — acceptable and matches the
  existing pattern.)

Notes:
- `ConnectionException` on a genuine connect failure / total-timeout will still
  propagate (and fail the job) — this is the agreed behavior (no retry changes).
  A 404 is surfaced as `$response->failed()` → empty generator → timestamp still
  stamped, unchanged.
- Do NOT touch `retrieveRebrickableDataFromUrl()`.

### 3. Stream-attach in `ImportColorLdrawImagesJob` (bounded memory)

File: `app/Domain/Part/Jobs/ImportColorLdrawImagesJob.php`

**Problem with current code:** it buffers ALL matched PNG bytes for a color into
`$pngsByStem` (an `array<string, string>`) before resolving part ids and
attaching. For a ~700 MB archive that is a large memory footprint even after the
download itself is streamed to disk.

**Refactor:** process ZIP entries in **bounded batches** so at most one batch's
worth of PNG bytes is in memory at a time, while still resolving part ids with a
single `whereIn` query per batch (not per entry).

Design:

- Add a batch-size constant, e.g.:

  ```php
  /**
   * How many PNG entries to buffer before resolving part ids and attaching,
   * bounding peak memory for very large color archives.
   *
   * @var int<1, max>
   */
  protected int $attachBatchSize = 200;
  ```

- Stream entries and accumulate into a small buffer keyed by stem; when the
  buffer reaches `$attachBatchSize`, flush it (resolve + attach), then clear it:

  ```php
  $color = Color::find($this->colorId);

  if ($color === null) {
      return;
  }

  $url = sprintf((string) config('rebrickable.ldraw_parts_zip_url'), $color->rebrickable_id);

  $entriesSeen = 0;
  $matchedParts = 0;
  $attached = 0;

  /** @var array<string, string> $buffer  stem => png bytes */
  $buffer = [];

  foreach ($downloader->retrieveZipEntriesFromUrl($url) as $name => $bytes) {
      $filename = basename($name);

      if (! str_ends_with(strtolower($filename), '.png')) {
          continue;
      }

      $entriesSeen++;
      $buffer[substr($filename, 0, -strlen('.png'))] = $bytes;

      if (count($buffer) >= $this->attachBatchSize) {
          [$m, $a] = $this->attachBatch($color, $buffer);
          $matchedParts += $m;
          $attached += $a;
          $buffer = [];
      }
  }

  // Flush the remainder.
  if ($buffer !== []) {
      [$m, $a] = $this->attachBatch($color, $buffer);
      $matchedParts += $m;
      $attached += $a;
  }

  $color->update(['ldraw_images_imported_at' => now()]);

  Log::info('Imported LDraw images for color.', [
      'color_id' => $color->id,
      'rebrickable_id' => $color->rebrickable_id,
      'png_entries_seen' => $entriesSeen,
      'matched_parts' => $matchedParts,
      'images_attached' => $attached,
      'unknown_part_skips' => $entriesSeen - $matchedParts,
  ]);
  ```

- Extract the resolve+attach logic into a private helper that returns
  `[matchedCount, attachedCount]`:

  ```php
  /**
   * Resolve part ids for a batch of stems and attach each PNG to its
   * part_color in the LDraw collection. Returns [matched, attached].
   *
   * @param  array<string, string>  $pngsByStem  stem => png bytes
   * @return array{int, int}
   */
  private function attachBatch(Color $color, array $pngsByStem): array
  {
      /** @var SupportCollection<string, int> $partIds */
      $partIds = Part::query()
          ->whereIn('rebrickable_id', array_keys($pngsByStem))
          ->pluck('id', 'rebrickable_id');

      $matched = 0;
      $attached = 0;

      foreach ($pngsByStem as $stem => $bytes) {
          $partId = $partIds->get($stem);

          if ($partId === null) {
              continue;
          }

          $matched++;

          $partColor = PartColor::firstOrCreate([
              'part_id' => $partId,
              'color_id' => $color->id,
          ]);

          // Idempotent: keep an existing LDraw image rather than duplicating it.
          if ($partColor->getFirstMedia(PartColor::LDRAW_IMAGE_COLLECTION) !== null) {
              continue;
          }

          $partColor
              ->addMediaFromString($bytes)
              ->usingFileName("{$stem}-{$color->rebrickable_id}.png")
              ->toMediaCollection(PartColor::LDRAW_IMAGE_COLLECTION);

          $attached++;
      }

      return [$matched, $attached];
  }
  ```

Behavior preserved:
- Same filename → stem mapping, same `firstOrCreate`, same idempotency check,
  same collection + filename convention, same timestamp stamping on completion
  (including the empty-archive/404 path, which simply yields no entries and falls
  straight through to the `$color->update(...)`).
- `unknown_part_skips` is now derived as `entriesSeen - matchedParts` across all
  batches (equivalent to the previous whole-archive computation).

Peak memory is now bounded to ~`$attachBatchSize` PNGs instead of the entire
color archive.

`ImportPartColorImagesCommand` is unchanged.

# Part-Color Media Conversions (preview / medium / large)

## Goal

Part-color images are already stored as Spatie MediaLibrary media on the
`part_colors` pivot (the `ldraw` collection from the import, plus the legacy
`image` collection). This task updates the **registered conversions** so every
attached image gets three width-based variants suited for previews and larger
displays:

- `preview` — fits within **150 × 150**
- `medium` — fits within **350 × 350**
- `large` — fits within **500 × 500** (previously 600 × 600)

All three preserve aspect ratio (`Fit::Contain`), so square LDraw renders become
NxN and non-square images fit inside the NxN box without distortion. `large` is
explicitly converted to the 500 box even though most LDraw renders are already
≤500px (guarantees a consistent upper bound).

## Decisions (confirmed with user)

- **Sizing mode:** `Fit::Contain` to `(N, N)` for each conversion (fit within
  the NxN box, keep aspect ratio). Matches the "500x500 by default" wording.
- **Names:** keep `preview` (now 150, was 150) and `large` (now **500**, was
  600); add a new `medium` (350). This preserves the names existing consumers
  already reference, so no consumer changes are required.
- **Scope:** conversions stay **global** in
  `PartColor::registerMediaConversions()`, so they apply to BOTH the `image` and
  `ldraw` collections.

## Current State (for reference)

`app/Models/Pivots/PartColor.php::registerMediaConversions()`:

```php
$this->addMediaConversion('preview')
    ->fit(Fit::Contain, 150, 150);

$this->addMediaConversion('large')
    ->fit(Fit::Contain, 600, 600);
```

Consumers referencing conversion names (all on `IMAGE_COLLECTION`, unaffected by
this rename since names are preserved):
- `app/Http/Resources/ProductResource.php` → `'large'`
- `app/Http/Resources/ProductSearchResource.php` → `'preview'`
- `app/Services/CartService.php` → `'preview'`

Images are attached in `ImportColorLdrawImagesJob` via `addMediaFromString(...)
->toMediaCollection(PartColor::LDRAW_IMAGE_COLLECTION)` — conversions are
generated automatically by Spatie when media is added (queued by default).

## Implementation Steps

### 1. Update `PartColor::registerMediaConversions()`

File: `app/Models/Pivots/PartColor.php`

Replace the two conversions with three:

```php
public function registerMediaConversions(?Media $media = null): void
{
    $this->addMediaConversion('preview')
        ->fit(Fit::Contain, 150, 150);

    $this->addMediaConversion('medium')
        ->fit(Fit::Contain, 350, 350);

    $this->addMediaConversion('large')
        ->fit(Fit::Contain, 500, 500);
}
```

Notes:
- `Fit::Contain` preserves aspect ratio and scales the image to fit within the
  NxN box (Spatie/Glide default behavior). No canvas padding is added.
- No collection-name guard: conversions apply to both `image` and `ldraw` media
  (the intended global scope).
- `media_library` already persists generated conversions
  (`generated_conversions` column exists in
  `database/migrations/2026_05_30_134534_create_media_table.php`); no migration
  or config change needed.

### 2. (No changes) Storage / job / consumers

- The import job, command, download service, and media collection registration
  are unchanged.
- `ProductResource` (`large`), `ProductSearchResource` (`preview`), and
  `CartService` (`preview`) keep working because those conversion names are
  retained. `medium` is newly available for any consumer that wants it (no
  consumer is required to use it now).
- Existing media rows created before this change will only have the previously
  generated conversions until regenerated. Regenerating conversions for already
  imported media is OUT OF SCOPE here (can be done later via
  `php artisan media-library:regenerate` if/when needed). New imports get all
  three immediately.

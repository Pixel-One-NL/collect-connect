# Improved Images Import Plan

Goal: Pre-generate every part-color combination that we actually sell as a `Product`, import the LDraw ID into the `parts` table, and then match LDraw renders by `ldraw_id` instead of by `rebrickable_id` so we can attach an image to every relevant part-color pair (while keeping Spatie MediaLibrary as the storage layer so conversions can be added later).

---

## 1. Schema change: add `parts.ldraw_id`

- Add an index (`->index()`) — the LDraw image import will look parts up by `ldraw_id` per batch, and there can be multiple parts sharing the same `ldraw_id` (geometric duplicates), so it must NOT be unique.


### `app/Domain/Part/Jobs/ImportPartBricklinkNumbersJob.php`

Rename + broaden responsibilities while keeping it backwards compatible:

- Rename file/class to `ImportPartRebrickableExternalIdsJob` (or keep the name — see Open Question 1). The plan continues assuming we keep the class name to avoid touching schedulers and callers.
- Selection query (`retrievePartsToUpdate`):
  - Today: `whereNull('bricklink_id')` and resumes via `lastProcessedPartId` stored from the highest non-null bricklink id.
  - Change to: pick parts where **either** `bricklink_id IS NULL` OR `ldraw_id IS NULL`. Compute the resume cursor as the highest `id` among parts where **both** fields are filled. - Update step: in the `DB::transaction`, update both columns:
  ```php
  Part::query()
      ->where('rebrickable_id', $rebrickablePart->partNum)
      ->update([
          'bricklink_id' => $rebrickablePart->bricklinkIds?->first(),
          'ldraw_id'     => $rebrickablePart->ldrawIds?->first(),
      ]);
  ```
- Self-dispatch passes `lastProcessedPartId` as today.

## 3. Pre-generate `part_colors` from existing products

Until now `part_colors` rows only existed when an LDraw render happened to land on a known part. We now want a row for every (part, color) pair that has a `Product`.

### New command + service/job

- New artisan command: `php artisan part-color:sync-from-products`
  - File: `app/Domain/Part/Commands/SyncPartColorsFromProductsCommand.php`
  - Signature: `part-color:sync-from-products` (no options for now).
  - Behaviour: idempotently `INSERT ... SELECT` into `part_colors` every `(productable_id, color_id)` from `products` where `productable_type = (new Part)->getMorphClass()`. Use `INSERT IGNORE`/`upsert` against the existing `unique(part_id, color_id)` index so re-running is a no-op.
  - Implementation sketch:
    ```php
    DB::statement("
        INSERT IGNORE INTO part_colors (part_id, color_id)
        SELECT productable_id, color_id
        FROM products
        WHERE productable_type = ?
          AND color_id IS NOT NULL
    ", [(new Part)->getMorphClass()]);
    ```
  - Logs a summary line with `part_colors_total`.

- Hook the same logic into `ImportInventoryJob::handle()` (after `upsertProducts`) so newly-created products immediately produce `part_colors`. Extracted as a small private method or a shared `SyncPartColorsFromProducts` action class under `app/Domain/Part/Actions/` to avoid duplication.

### Why not Inventory or Cartesian?

Per the user: "Create them based off our own products table. When there's a product attached to a part, there should be a part_color entry." This matches our actual catalog, keeps the table bounded, and ensures every salable variant gets a render slot.

### Tests

- `tests/Feature/Domain/Part/SyncPartColorsFromProductsTest.php`:
  - Creates products (part + color) and asserts matching `part_colors` rows are present after the command/action.
  - Asserts idempotency: running twice does not produce duplicates and does not throw.
  - Asserts the inventory-import path also produces `part_colors` (use `Bricqer::fake`-style or the existing test pattern for `ImportInventoryJob`).

## 4. Switch LDraw image matching to use `ldraw_id`

Rebrickable's per-color LDraw archives (`parts_{color_rebrickable_id}.zip`) name their PNG files by **LDraw geometry id**, not by Rebrickable part id. Today `ImportColorLdrawImagesJob` matches by `Part.rebrickable_id` which only works for the parts where the two ids happen to coincide; switching to `ldraw_id` is what unlocks correct image attachment for the rest.

### `app/Domain/Part/Jobs/ImportColorLdrawImagesJob.php`

- In `attachBatch()`:
  - Change the lookup from `whereIn('rebrickable_id', array_keys($pngsByStem))` to `whereIn('ldraw_id', array_keys($pngsByStem))`.
  - `Part::query()->whereIn('ldraw_id', ...)->get(['id', 'ldraw_id', 'rebrickable_id'])` and key by `ldraw_id`. Because multiple parts can share an `ldraw_id` (variants, prints), iterate and attach the same PNG bytes to each matching part's `part_color`.
- Filename written via `usingFileName()` — keep the `"{stem}-{color}.png"` shape but use the LDraw id stem; this keeps storage stable per archive.
- Variant fallback (`bufferVariantFallbacks` / `attachFallback`):
  - The current logic strips a trailing letter (`30151b` → base `30151`) using `rebrickable_id`. Re-base it on `ldraw_id` for both the candidate generation and the `substr($variant->ldraw_id, 0, -1)` step. Parts with a null `ldraw_id` are simply ignored by this pass (their LDraw render cannot be located).
- The `firstOrCreate` in `attachImage()` stays the same — `part_colors` are now pre-populated by step 3, but `firstOrCreate` preserves backwards compatibility for any pair that wasn't pre-seeded.

### `app/Domain/Part/Commands/ImportPartColorImagesCommand.php`

- Add a prerequisite step in `handle()`: run / dispatch the `SyncPartColorsFromProducts` action before queueing the per-color jobs, so the pre-seeded pairs exist by the time images attach. Behind a `--skip-sync` flag if we want to keep the existing behaviour available for tests.

### Tests

Update `tests/Feature/Domain/Part/ImportColorLdrawImagesTest.php`:

- Replace `Part::factory()->create(['rebrickable_id' => '3001'])` with the same factory but also supplying `ldraw_id => '3001'` (or a deliberately different value to prove `ldraw_id` is what's matched).
- Add a new test: a part with `rebrickable_id = '3001'` but `ldraw_id = '3001a'` — the archive contains `3001a.png`, not `3001.png`, and the test asserts the image is attached.
- Add a new test: a part with `ldraw_id = null` — the archive's PNG must not be attached to it.
- The existing variant fallback tests are updated to use `ldraw_id` for the variant suffix logic.

## 5. Order of operations / scheduling

Document (no code) and adjust callers where needed:

1. Run `php artisan rebrickable:import-entity` (populates `parts`, `colors`, `products`-feeding inventories).
2. Run `php artisan rebrickable:import-entity --entity=inventory_parts` etc. as today.
3. Run the existing Bricqer-driven `product:import-inventory` (creates `products`).
4. Run `php artisan parts:import-bricklink-numbers` (now also fills `ldraw_id`). Verify the existing command file name (`grep -R 'ImportPartBricklinkNumbersJob' app/Console`) — if there isn't a command yet, add a thin one under `app/Domain/Part/Commands/ImportPartBricklinkNumbersCommand.php` to dispatch the job manually for backfills.
5. Run `php artisan part-color:sync-from-products` (idempotent).
6. Run `php artisan part:import-color-images` (calls step 5 first if `--skip-sync` is not passed).

`ImportInventoryJob` also calls the part-color sync action inline so the chain "new products → new part_colors" stays automatic between scheduled runs.

# Bricqer image fallback plan

Goal: for every `part_color` that has no LDraw render after the existing import, attach the part's Bricqer "lego-visual" image into a new, dedicated media collection — without touching the existing `rebrickable_img_url` fallback (kept as a second safety net).

The Bricqer `lego-visual/item` endpoint cannot be filtered by `rebrickable_id` and ships ~95k rows. The plan therefore splits the work in two:

1. A **one-shot URL walk** that persists the per-part image URL onto the `parts` table.
2. A **per-color attach pass** that only downloads bytes for `part_colors` that actually need one (LDraw collection empty).

(Same shape as the prior `rebrickable_img_url` work; not unique, not indexed — only used as a per-part lookup once the row is loaded.)

## 2. Bricqer integration: `LegoVisualResource` + `ListItemsRequest`

The endpoint:

```
GET https://collect2connect.bricqer.com/api/v1/lego-visual/item?part_type=P[&page=N]
```

Response shape (see prompt): paginated; each result has `partNumber`, `partName`, `defaultColorId`, `image`, `partType`. We only care about `partNumber` + `image` for now.

### New files

- `app/Integrations/Bricqer/Resources/LegoVisualResource.php` — exposes `items(): LegoVisualItemsResource`.
  - `BricqerApi::legoVisual(): LegoVisualResource` added (mirrors `lego()` / `inventory()`).
- `app/Integrations/Bricqer/Resources/LegoVisual/LegoVisualItemsResource.php` — chainable resource with `partType(string $type): self` and `get(int $page = 1)` returning the page DTO. Builds a `ListItemsRequest`.
- `app/Integrations/Bricqer/Requests/LegoVisual/ListItemsRequest.php`:
  - `Method::GET`, endpoint `/lego-visual/item`.
  - `defaultQuery()` returns `array_filter(['part_type' => $partType, 'page' => $page])`.
  - `defaultHeaders(): ['Accept' => 'application/json']`.
  - `createDtoFromResponse(Response $response): LegoVisualPage` — populates a `LegoVisualPage` DTO from `response.json()`.
- `app/Integrations/Bricqer/DataTransferObjects/LegoVisual/LegoVisualItem.php` — Spatie `Data` DTO:
  - `int $id`
  - `#[MapName('partType')] string $partType`
  - `#[MapName('partNumber')] string $partNumber`
  - `#[MapName('partName')] ?string $partName`
  - `#[MapName('defaultColorId')] ?int $defaultColorId`
  - `?string $image`
- `app/Integrations/Bricqer/DataTransferObjects/LegoVisual/LegoVisualPage.php` — Spatie `Data` DTO:
  - `int $number` (mapped from `page.number`)
  - `int $size` (mapped from `page.size`)
  - `int $count` (mapped from `page.count`)
  - `?string $nextUrl` (mapped from `page.links.next`)
  - `/** @var DataCollection<int, LegoVisualItem> */ public DataCollection $results`
  - Static `fromResponse(array $json): self` builds it (Spatie `Data` handles nested mapping with `#[MapName]` paths like `page.links.next`).
- `app/Integrations/Bricqer/Resources/LegoVisual/` directory created alongside `Lego/`.

> Convention check: the existing `GetColorsRequest` uses `Saloon\Http\Request` (not `PaginatedRequest`). Bricqer's pagination is custom (no `page_size` query param needed; just `page=`), so a dedicated request class — not a shared `PaginatedRequest` — keeps it explicit and avoids leaking Rebrickable conventions into Bricqer.

## 3. Job + command: walk all pages, fill `parts.bricqer_img_url`

### `app/Domain/Part/Jobs/ImportPartBricqerImageUrlsJob.php` (new)

- `ShouldQueue`, `Dispatchable`, `Queueable`, `InteractsWithQueue`.
- Constructor: `public ?int $page = 1` so the job is self-dispatching per page (mirrors `ImportPartBricklinkNumbersJob`'s resumability).
- `handle(BricqerApi $bricqer): void`:
  1. Fetch page via `Bricqer::legoVisual()->items()->partType('P')->get($this->page)`.
  2. Collect rows into `$updates = ['lowercased_partNumber' => $image, …]`, skipping rows whose `image` is null.
  3. Lookup parts in one query:
     ```php
     Part::query()
         ->whereIn('bricklink_id', array_keys($updates))
         ->select(['id', 'bricklink_id'])
         ->get()
         ->groupBy(fn (Part $p) => strtolower($p->bricklink_id));
     ```
     (Case-insensitive match, matching the same approach used in `ImportInventoryJob::upsertProducts()`.)
  4. For each match, `Part::query()->where('id', $id)->update(['bricqer_img_url' => $url])`. Wrapped in a single `DB::transaction()` per page.
  5. If `$page->nextUrl !== null`, self-dispatch with `$this->page + 1` (with `->delay(1)` to be polite to Bricqer, matching how `ImportPartBricklinkNumbersJob` self-throttles).
  6. Log per-page stats: `{page, rows, matched, updated}`. Final-page logs a summary.
- Tests asserting:
  - One page worth of rows: matched parts get `bricqer_img_url` set, unmatched rows ignored, case-insensitive matching works.
  - Multi-page: paging via `Saloon::fake` with two `MockResponse`s (page=1 returns `page.links.next` non-null; page=2 returns `next=null`).
  - Idempotent: rerunning leaves the same values.

### `app/Domain/Part/Commands/ImportPartBricqerImageUrlsCommand.php` (new)

- Signature: `part:import-bricqer-image-urls {--page=1 : Start from a specific page}`.
- `handle()`: dispatches `ImportPartBricqerImageUrlsJob` with the given starting page. No interactive prompts (per Boost guidelines).

## 4. Per-color attach pass: Bricqer image bytes -> `BRICQER_IMAGE_COLLECTION`

### `app/Models/Pivots/PartColor.php`

Add a third media-collection constant:

```php
public const string BRICQER_IMAGE_COLLECTION = 'bricqer';
```

Register it as a single-file collection alongside the existing two:

```php
$this->addMediaCollection(self::BRICQER_IMAGE_COLLECTION)->singleFile();
```

Keeps Bricqer images separate from LDraw and Rebrickable images so display logic can prioritise (LDRAW > BRICQER > IMAGE) later.

### Reuse the existing downloader

`RebrickableDownloader::retrieveFileFromUrl()` already does exactly what we need (HTTP GET, returns bytes or `null`). We rename / repurpose it OR add a sibling contract to keep semantics clean:

- **Decision**: keep `RebrickableDownloader::retrieveFileFromUrl()` as-is — its implementation is provider-agnostic (it does not include any Rebrickable-specific auth). The Bricqer images are served from a public CDN (`cdn.bricqer.com`), so no auth header is needed. We pass the URL directly to the same method. A small docblock tweak (mentioning that it's also used for Bricqer CDN URLs) is enough.

If, during implementation, we discover the CDN requires auth or different timeouts, we'll extract a generic `HttpFileDownloader` interface in a follow-up — flagged but not required up front.

### `app/Domain/Part/Jobs/ImportColorBricqerImagesJob.php` (new)

Mirrors the structure of `ImportColorLdrawImagesJob::attachRebrickableImageFallback()` but is its own job so progress per-color is independent and batchable.

- `ShouldQueue`, `Batchable`, `Dispatchable`, `Queueable`, `InteractsWithQueue`.
- Constructor: `public int $colorId`.
- `handle(RebrickableDownloader $downloader)`:
  1. Resolve the `Color`.
  2. Query:
     ```php
     PartColor::query()
         ->where('color_id', $color->id)
         ->whereHas('part', fn (Builder $q) => $q->whereNotNull('bricqer_img_url'))
         ->whereDoesntHave('media', fn (Builder $q) => $q->where('collection_name', PartColor::LDRAW_IMAGE_COLLECTION))
         ->with('part:id,bricqer_img_url')
         ->chunkById(200, …);
     ```
     i.e. **only** part_colors lacking an LDraw render, whose part has a `bricqer_img_url`. (Per the user's "Bricqer-only fallback" answer: we do NOT defer to `rebrickable_img_url`. The existing rebrickable fallback path continues to run independently in `ImportColorLdrawImagesJob`.)
  3. For each row: skip if a Bricqer image is already attached (idempotent). Download bytes (caching per URL within the run, same as the existing pattern). Attach to `BRICQER_IMAGE_COLLECTION`.
  4. Log `{color_id, considered, attached, download_failures}` at the end.
- Adds a `bricqer_images_imported_at` column on `colors` so we can apply a freshness window like the LDraw one. Migration: `colors.bricqer_images_imported_at` (timestamp, nullable). `ColorFactory::bricqerImagesImportedAt()` state helper added analogously to `ldrawImagesImportedAt()`.
- The job stamps the timestamp on success; leaves it null when the download fails completely, so the next run retries.

### `app/Domain/Part/Commands/ImportPartColorBricqerImagesCommand.php` (new)

- Signature: `part:import-bricqer-images {--full} {--color=*}`.
- Mirrors `ImportPartColorImagesCommand` (without the SyncPartColorsFromProducts step — the part_color rows already exist by the time anyone runs Bricqer; if not, this command is a no-op for them, which is the safe behaviour).
- Selects colors via the same freshness-window pattern (`bricqer_images_imported_at` <= now()-7d or null).
- `Bus::batch($jobs)->allowFailures()->dispatch()`; `finally()` callback logs how many part_colors now have a Bricqer image.

### Tests for the per-color job

`tests/Feature/Domain/Part/ImportColorBricqerImagesTest.php`:

- Uses the same fake `RebrickableDownloader` already created in `ImportColorLdrawImagesTest` (it now serves Bricqer-CDN URLs too — the implementation is URL-agnostic).
- Cases:
  - Part with `bricqer_img_url` set and no LDraw render → image attached to `BRICQER_IMAGE_COLLECTION`.
  - Part with `bricqer_img_url` set but an existing LDraw render → no Bricqer attach.
  - Part with `bricqer_img_url = null` → skipped.
  - Idempotency: rerun does not duplicate the media.
  - Multiple part_colors share the same `bricqer_img_url` (different parts pointing to the same image, unusual but possible) → bytes downloaded once, attached to each pivot (asserted via single-download cache).
  - Command freshness: stale color queued, fresh color not.
  - Command `--full` ignores the freshness window.
  - Command `--color=` filter restricts colors.
  - Download failure leaves `bricqer_images_imported_at` null so the next run retries.

`tests/Feature/Domain/Part/ImportPartBricqerImageUrlsJobTest.php`:

- Uses `Saloon::fake([ListItemsRequest::class => MockResponse::make(...)])` for one and two-page scenarios.
- Asserts case-insensitive `bricklink_id` matching, unmatched rows ignored, and the next-page request is sent only when `page.links.next` is non-null.

## 5. Orchestration / order of operations

Document in the implementation summary (no code change required beyond commands):

1. `php artisan rebrickable:import-entity` (parts, colors)
2. `php artisan color:import-bricker` (maps bricker_color_id)
3. `php artisan product:import-inventory` (creates products → triggers part_color sync)
4. `php artisan part:import-bricklink-ids` (fills bricklink_id, ldraw_id, rebrickable_img_url)
5. `php artisan part-color:sync-from-products` (idempotent; usually redundant after step 3)
6. `php artisan part:import-color-images` (LDraw zip + rebrickable_img_url fallback)
7. **NEW** `php artisan part:import-bricqer-image-urls` (one-shot walk; fills parts.bricqer_img_url)
8. **NEW** `php artisan part:import-bricqer-images` (downloads bytes for part_colors still missing an LDraw render)

Steps 7 & 8 are additive; nothing about the existing import flow needs to change.

## 7. Risk / open notes (non-blocking)

- **CDN auth**: the prompt's example URL (`https://cdn.bricqer.com/static/bl-cache/PN/0/spa0021.png`) looks like an unauthenticated CDN path. If at implementation time the request fails 403/401, we'll add a `Saloon::send(GetImageRequest)` flavoured download instead. Plan documents this as a known unknown.
- **Walking 95k+ rows is long**: at 100 per page the job needs ~953 pages. The 1-second delay between self-dispatched jobs means ~16 minutes wall-clock, single-threaded. Acceptable for a once-a-week refresh; the `--page=` flag lets us resume mid-run.
- **`defaultColorId`**: present in the response but unused for now — the user wants every color of a salable part to get the same Bricqer image (Bricqer's image is not color-specific; it's a generic catalog photo). Documented; not implemented. Easy follow-up if the desired behaviour is "only attach to the part_color whose color matches `defaultColorId`".
- **Renaming `RebrickableDownloader`**: if Bricqer ends up needing different timeouts/auth, the natural follow-up is to extract a generic `HttpFileDownloader` interface. For now the contract stays under `Rebrickable\Contracts` — re-naming is purely cosmetic and not worth the churn until we have a second reason.
